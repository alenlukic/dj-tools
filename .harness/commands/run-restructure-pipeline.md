# Run Restructure Pipeline

DEVDSL-1
MODE: STRICT
FLAGS: NO_EARLY_STOP TEST_GATE(full) SCOPE_LOCK(explicit) OUTPUT_SCHEMA(default)

## COMMAND

Run the repo-local restructure pipeline for a scoped codebase restructuring task.

## INPUT

Required:
- `task`: plain-English description of the restructuring goal and scope

## SCOPE

Execute one scoped restructure run for the provided task using the repo-local pipeline.

This workflow is for structural improvement, not feature delivery.
Preserve existing behavior unless the task explicitly requires behavioral change.

Do not expand scope beyond the stated restructure boundary unless required to unblock correctness or complete an in-scope migration safely.

## DELEGATION

Each numbered step that names an agent must be delegated via `Task(subagent_type="<Agent Name>")`.
You are the orchestrator — do not perform agent work directly.
Pass the run directory path and relevant artifacts as context to each subagent.

## DO

1. Initialize run
- create a new run directory under `.harness/runs/`
- establish the active run context for this task

2. Intake and restructure planning
- use the `Delivery Supervisor` agent
- restate the restructure task into:
  - explicit scope boundaries
  - behavior-preservation expectations
  - acceptance criteria
  - likely relevant files/modules/tests
- write:
  - `TASK.md`
  - `PLAN.md`

3. Restructure analysis and implementation
- delegate to the `Restructure Coder`
- require the agent to:
  - understand current behavior before editing
  - analyze architectural and organizational issues
  - strengthen test coverage where needed
  - consider alternative structural approaches
  - iteratively port the implementation in safe slices
  - write `RESTRUCTURE_ANALYSIS.md`
  - write `RESTRUCTURE_SUMMARY.md`

4. Specific review loop
- delegate review to the `Delivery Reviewer` agent
- use this review to confirm:
  - behavior preservation
  - logical correctness
  - requirements adherence
  - no regressions introduced by the restructure
- if verdict is `CHANGES_REQUESTED`:
  - return findings to the `Restructure Coder`
  - perform focused remediation
  - repeat specific review within pipeline limits

5. QA validation
- delegate validation to the `Delivery QA` agent
- if QA verdict is `FAIL`:
  - return findings to the `Restructure Coder`
  - perform focused remediation
  - re-run the `Delivery Reviewer` as needed
  - repeat QA within pipeline limits

6. Broad review
- only after QA verdict is `PASS`, delegate review to the `Delivery Broad Reviewer` agent
- assess whether the restructure materially improves:
  - design quality
  - maintainability/extensibility
  - readability/organization
  - repo pattern alignment
  - longer-term change cost
- if broad review verdict is `CHANGES_REQUESTED`:
  - return findings to the `Restructure Coder`
  - perform focused remediation
  - re-run the `Delivery Reviewer`
  - re-run `Delivery QA`
  - after QA returns `PASS`, re-run the `Delivery Broad Reviewer`
  - repeat within pipeline limits

7. Verification and artifacts
- when running verification or collecting artifacts, use:
  - `python3 .harness/bin/pipeline.py`

8. Build verification
- delegate to the `Delivery Build Verifier` agent
- confirm the build process succeeds after all restructure changes
- if build status is `FAIL`:
  - return findings to the `Restructure Coder`
  - perform focused remediation
  - re-run the `Delivery Reviewer`
  - re-run `Delivery QA`
  - after QA returns `PASS`, re-run the `Delivery Broad Reviewer` as needed
  - repeat build verification within pipeline limits

9. Finalize
- end with a concise final summary focused on restructure outcomes

## VALIDATION

Before completion, verify:
- a run directory was created
- `TASK.md` and `PLAN.md` exist
- restructure analysis and implementation were executed by the `Restructure Coder`
- `RESTRUCTURE_ANALYSIS.md` and `RESTRUCTURE_SUMMARY.md` exist
- specific review, QA, broad review, and build verification were all executed
- specific review confirmed behavioral safety and requirement fit
- broad review occurred only after QA passed
- remediation after failures re-entered the specific review + QA path before re-running broad review
- verification/artifact collection used the pipeline runner
- build verification produced PASS or FAIL
- final summary reflects actual outcomes

## OUTPUT

Produce:
- active run directory under `.harness/runs/`
- delivery artifacts generated by the pipeline
- `RESTRUCTURE_ANALYSIS.md`
- `RESTRUCTURE_SUMMARY.md`
- `SPECIFIC_REVIEW_NOTES.md`
- `BROAD_REVIEW_NOTES.md`
- `BUILD_VERIFICATION.md`
- concise final summary including:
  - verdict
  - scope restructured
  - changed files
  - tests run
  - specific review status
  - QA status
  - broad review status
  - build status
  - unresolved caveats, if any

## ACCEPTANCE

Complete only if:
- the Delivery Supervisor coordinated the workflow
- context remained narrow and scope-locked
- the restructure agent analyzed behavior before editing
- test coverage was strengthened where needed for safe restructuring
- migration was iterative rather than a blind rewrite
- specific review validated correctness and behavior preservation
- QA validation was performed
- broad review was performed only after QA passed
- verification used `python3 .harness/bin/pipeline.py`
- build verification was performed and passed
- any failures were either resolved or explicitly documented
- final summary is evidence-backed
